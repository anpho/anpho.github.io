🛠️ 第一步：揪出嫌疑人（从软件列表分析）

虽然列表干净，但有几个东西会深度注入文件资源管理器：

  1. Git (Git.Git)：它默认会安装 Git Bash Here 和图标状态缓存（Git LFS/Overlay Handler）。当你在资源管理器进入一个包含巨大 git 仓库的目录，或者网络盘时，Git 的外壳扩展在后台扫描文件可能会导致该窗口所在的线程直接卡死。
  2. 7-Zip (7zip.7zip)：拥有右键菜单扩展。虽然 7-Zip 极度稳定，但偶尔会和某些特定格式的文件预览冲突。
  3. 豹图 版本 9.3：这是一个 CAD/图纸查看工具。它是高度嫌疑对象。这类看图、专业制图软件为了方便用户，往往会强行注入资源管理器的“缩略图预览”和“右键菜单”。如果它在解析某个损坏的图纸缩略图，或者后台授权检测卡死，就会直接带崩当前的资源管理器窗口。
  4. Free Download Manager：会接管浏览器和系统的下载监控，有时会触发文件锁定。

🔍 第二步：精准捕获死锁（高级排查思路）

既然你已经有 Sysinternals Suite，下次当这个假死窗口再次出现时，千万别关掉它，按照以下方法抓现行：

1. 使用 Process Explorer 抓取线程

  1. 以管理员身份运行 procexp.exe。
  2. 找到 explorer.exe（如果开了“在单独的进程中打开文件夹窗口”，可能会有多个，找 CPU 或内存有异常波动的那个）。
  3. 双击 explorer.exe 打开属性页，切换到 Threads（线程） 选项卡。
  4. 按 CPU 或 State 排序，寻找处于等待状态（如 Wait:UserRequest 且有卡死迹象）的线程。
  5. 看一下这些卡住的线程的 Start Address（起始地址）。如果看到类似 BaoTuShellExt.dll、git_shell_ext.dll 或 7-zip.dll，恭喜你，直接破案。

2. 用 WinDbg 抓取转储文件（Dump）

如果你想彻底看清是哪个函数死锁了：

  1. 假死发生时，打开任务管理器，切换到“详细信息”选项卡。
  2. 右键点击假死的 explorer.exe -> 创建转储文件。
  3. 用 WinDbg 打开这个 .dmp 文件。
  4. 输入命令 !analyze -v 让它自动分析，或者输入 ~*k 查看所有线程的调用栈，寻找卡在 WaitForSingleObject 或某个第三方 DLL 上的线程。

🚀 另外两个极高概率的“暗桩”

在 Win10 19044 (21H2) 这个版本中，特定单窗口假死最常见的两个暗桩：

1. 幽灵网络/快照路径超时（网络 I/O 死锁）

这是 Windows 资源管理器的老无赖病了。

  • 现象：当资源管理器窗口视图试图去读取一个“不存在”或者“正在休眠/断开”的路径时，负责 UI 刷新的主线程会强行等待网络 I/O 回应。由于超时时间（Timeout）通常长达几十秒甚至几分钟，在此期间，该窗口内容区完全不响应任何点击，但任务栏和窗口边框（缩放）由 DWM（窗口管理器）托管，所以依然可动。
  • 排查：
  • 该窗口假死前，你是不是刚点过“此电脑”？或者它的侧边栏是否有无法访问的 NAS、网络共享文件夹、断开的手机多屏协同盘符？
  • 检查注册表路径 HKEY_CURRENT_USER\Network 和 HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2，看看有没有很久以前挂载过、现在已经废弃的临时网络盘残留，如果有，直接删掉对应项。

2. 缩略图/预览窗格死锁（COM 组件崩溃）

  • 现象：如果你开启了“预览窗格”或者文件夹里有大文件（特别是 CAD 图纸、特殊格式视频），资源管理器会调用对应的 COM 组件（Handler） 去解析文件提取缩略图或预览。如果这个组件（比如“豹图”的图纸解析器）卡死在内核态，对应的资源管理器窗口就会永久失去响应。
  • 验证与解决：
  1. 在 Autoruns 中，切换到 Everything 选项卡。
  2. 在上方过滤器（Filter）中输入 Preview（预览）或者 PropertyHandler（属性处理器）、Thumbnail（缩略图）。
  3. 看看有没有和“豹图”或第三方看图/制图软件相关的条目，如果有，那就是它们在作祟。

🔬 堆栈行为深度解析

这个线程是 explorer.exe 的UI主线程（或者该独占窗口的主线程）。

  1. CExplorerHostCreator::RunHost 正在尝试创建或初始化资源管理器的主窗口宿主。
  2. 它调用了 SHProcessMessagesUntilEventsEx，这意味着它在等待某些异步事件/消息的返回，在这之前它必须挂起 UI。
  3. 它最终卡在了 NtUserMsgWaitForMultipleObjectsEx（无响应的根本原因）。这个函数是一个阻塞调用。它在说：“我正在等另一个线程或者另一个进程给我发信号/回消息，在它回应我之前，这个窗口的 UI 消息循环拒绝处理任何点击。”

因为这里没有看到明显的第三方 DLL 崩溃或直接报错，它是一个典型的逻辑死锁（Deadlock）或永久超时等待。这非常符合你描述的“窗口能最小化/关闭（由 DWM 托管的非客户区），但内容区完全失去响应”的症状。

由于这里是微软的原生调用栈，这意味着嫌疑人大概率在另外一个工作线程里，或者由于某种环境卡住了主线程等待的那个 Event。

💡 基于此堆栈的高概率原因推测

在 CExplorerHostCreator::RunHost 阶段卡死，通常和以下三者强相关：

  1. 快速访问（Quick Access）中的网络/幽灵路径死锁
    当窗口刚创建，RunHost 准备加载视图，它会去读取“快速访问”或者侧边栏的图标。如果里面残留了一个不可达的网络路径（如已经断开的 NAS 共享、WPS云盘残余、手机无线连接盘），主线程发出的查询请求被卡在内核的 I/O 队列里，导致 Events 永远不会被触发。
  2. 多显示器 / 显卡驱动的钩子挂起
    你安装了 Intel 核心显卡驱动和 Arc 显卡软件。有时候显卡控制面板的某些“窗口管理”、“智能外壳增强”功能会挂钩新创建的 Explorer 窗口。如果它在计算窗口位置或渲染特效时没有回应，主线程也会卡死在这里。
  3. 特定的 Shell Folder 注册表损坏
    Windows 尝试重定向某些核心文件夹（如“下载”、“文档”）到不存在的路径，导致初始化失败。

🔬 案情复盘：4号线程在干嘛？

我们来看 4号线程 的核心堆栈：

💡 核心逻辑：

  1. 0号线程（UI主线程）发出了一个创建/渲染窗口主体的请求（RunHost），然后挂起自己，等待后台工作线程完成。
  2. 负责执行具体文件系统任务的是 4号线程。它调用了 windows_storage!CShellTask，代表它正在试图加载某个特定的“外壳文件夹”（Shell Folder）。
  3. 紧接着，4号线程派生出了一个独立的浏览器窗口消息泵 CExplorerFrame::FrameMessagePump，并被挂起在 NtUserWaitMessage。
  4. 结论：4号线程在试图枚举、读取或渲染某个文件夹（通常是“此电脑”里的非标准盘符、虚拟盘符或特定的系统已知文件夹）时，被操作系统内核发出的某个 IO 或 RPC 请求卡死在内核态（所以没进应用层的错误捕获）。由于它不返回，0号线程（主线程）只能永远死等。

🛠️ 针对性解决思路

既然堆栈已经100%锁定了是 windows_storage 在处理特殊系统文件夹时卡死，结合你的 winget 干净列表，我们需要对以下三个“暗桩”进行定向爆破：

1. 斩断远程与云端组件（最致命的嫌疑人：Office 2024）

注意看，你安装了 Office 2024 LTSC。Office 安装后会强行在“此电脑”里塞入一个特殊的文件夹外壳组件（例如 OneDrive 快捷方式、或者 SharePoint 团队网站的虚拟映射）。
即使你不用它，windows_storage 在创建新窗口、扫描路径时，都会去尝试通信这个通过 COM 注册的特殊文件夹（即你 Autoruns 里看到的 MSOXMLMF.DLL 和 MSOSB.DLL 托管的内容）。如果通信断开，4号线程就会直接挂死在 WaitMessage。

  • 解决办法：清理“此电脑”里的幽灵图标。
    打开注册表（regedit），导航到：
    HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\MyComputer\NameSpace
    这里面的每个子项（一串大括号 {...}）代表“此电脑”里显示的一个特殊系统文件夹。除了微软默认的“下载/文档/图片”等，如果有指向 Office、OneDrive 或者已卸载软件的残留，直接将其右键导出备份后删除。

2. 重置“文件资源管理器”的启动点

因为卡在 CExplorerHostCreator::RunHost 阶段，说明只要一开窗口就触发。我们必须打破它一启动就去读取“坏掉的缓存/坏掉的路径”的循环。

  • 打开任意文件夹 -> 点击上方“查看” -> “选项”。
  • 将“打开文件资源管理器时打开”改为 此电脑（如果是“快速访问”，它每次启动都会让 windows_storage 去扫描一堆历史文件和网络路径，极易引发 4号线程卡死）。
  • 并在下方隐私选项中，取消勾选“显示最近使用的文件”和“显示常用文件夹”，并点击清除。

3. 清理已断开的“幽灵”网络映射/凭据

如果你曾经用 Git 挂载过远程、或者 Free Download Manager 尝试向某个局域网/共享目录写入过文件，Windows 会在后台记住这个路径。

  • 检查注册表路径：
    HKEY_CURRENT_USER\Network
    HKEY_CURRENT_USER\SOFTWARE\Microsoft\Windows\CurrentVersion\Explorer\MountPoints2
    如果在这里看到任何已经不再使用的网络共享 IP 或盘符，全部删掉。

🏁 总结

这个假死问题是由 Windows 10 的 windows_storage.dll 在渲染特定系统/重定向文件夹时，后台 IO 被无限期挂起 导致的。

最快见效的实操：

  1. 进注册表把 NameSpace 里非必要的第三方大括号删掉（特别是带 Office/云盘性质的）。
  2. 把资源管理器改为默认打开“此电脑”并关闭历史记录隐私。

改完后重启 explorer.exe，这个折磨你许久的单窗口假死基本就能绝迹了！

这群线程是 windows_storage 派出去专门读取文件系统属性、解析图标、获取缩略图或查询磁盘状态的。它们都在等一个信号，但谁也没等到。

而在所有罢工的工人中，9号线程（Id: 1dec.3b80） 暴露了最核心的线索。

这证实了我们在注册表里删掉的只是常规干扰项。资源管理器在启动这短短 46 秒内，9号线程在试图拉起一个外壳扩展组件（Shell Extension）。这个组件本身拥有自己的窗口或消息循环（触发了 CExplorerFrame::FrameMessagePump），但它在初始化时彻底僵死了。

因为它是通过 COM 组件内联（In-process）加载到 explorer.exe 内部的，它不返回，不仅导致 0号主线程在 RunHost 永远死等，还把那十几个负责读取文件信息的工作线程全卡在内核队列里，形成了连锁假死。

既然不是注册表里的特殊文件夹，那它必然是一个潜伏在右键菜单、文件预览、或图标覆盖（Icon Overlay）中的第三方扩展 DLL。

🔍 终极破案：两大系统内置嫌疑犯

在你的模块列表里，有三个非常特殊的系统自带扩展被加载了：

  1. dlnashext.dll (DLNA Media Servers Shell Extension)
  2. wpdshext.dll (Portable Devices Shell Extension)
  3. davclnt.dll / davhlpr.dll (WebDAV 客户端核心)

它们分别对应：网络上的 DLNA 流媒体服务器（如电视、局域网共享播放器）、数据线连接的便携设备（如手机、相机），以及 WebDAV 网络网盘。

为什么是它们？

你的 9 号线程之所以卡在 win32u!NtUserWaitMessage 消息泵，是因为这些组件在资源管理器启动 46 秒内，主动拉起了一个后台线程去扫描局域网内的网络设备、或通过 WebDAV 握手远程服务器。

  • 如果你的局域网里有一个“半死不活”的流媒体设备（比如断网的智能电视、不响应的路由器 NAS 共享），或者你曾连接过一个现在已经无法访问的 WebDAV/网络位置，微软这个内置的 dlnashext 或 davclnt 就会死等网络 socket 握手，直接把整个资源管理器的全局调度器（CShellTaskScheduler）死锁掉！
